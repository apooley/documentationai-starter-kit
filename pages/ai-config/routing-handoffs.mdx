## How routing rules and handoff settings work

Routing Rules define **when** conversations are automatically transferred to specialized assistants based on keywords, intent, channel, and priority. Handoff Settings define **how** those transfers execute, including the message the customer sees and guardrails that prevent excessive or looping handoffs.

Together, they create a complete handoff system:

- Routing Rules match conversations that should be routed.
- Handoff Settings apply guardrails and templates.
- Approved handoffs send a structured message and transfer ownership.

<Callout kind="info">

You need tenant admin or equivalent configuration permissions to edit routing rules and handoff settings. Coordinate with support and operations leads before changing production behavior.

</Callout>

## End-to-end handoff flow

The routing and handoff engine evaluates every transfer request in a fixed order:

1. A Routing Rule matches the conversation based on triggers.
2. Handoff guardrails in Handoff Settings run checks.
3. If all checks pass, the system sends the handoff message and transfers the conversation to the target assistant or human queue.

If a guardrail blocks the transfer, the current assistant or human remains in control and can choose another action.

## Configure routing and handoffs

Use this workflow to configure routing rules and handoff settings safely before they affect live traffic.

<Steps>

<Step title="Map your routing and handoff goals" icon="settings">

Start by defining how you want conversations to move between assistants and humans.

- List your primary channels (SMS, chat, voice, email) and where each should land first.
- Identify specialized topics or intents (for example, Sales, Support, Billing, Technical) that need dedicated assistants.
- Decide when AI must escalate to humans by policy and when humans can hand work back to AI.
- Capture at least one default route and one escalation path for each major channel.

You should end this step with a clear sketch of which assistants and teams handle which intents, channels, and segments.

</Step>

<Step title="Prepare target assistants and queues" icon="zap">

Create or refine the destinations that routing rules will use.

- Use **Assistants** to define configurations for general support, sales, billing, technical help, and other roles.
- Create variants when you want to experiment on a subset of traffic.
- Align each assistant with the channels it should support and any playbooks it will run.

Confirm that each target assistant behaves as expected in its own tests before you send routed traffic to it.

</Step>

<Step title="Configure routing rules in the UI" icon="code">

Open the **Routing Rules** tab and define when conversations should be transferred.

- Add a rule for each common combination of trigger and destination (for example, Billing intent → Billing assistant on chat and email).
- Set priorities so more specific rules override broader ones.
- Use statuses to control whether a rule is live or parked for later.

Use internal test conversations to verify which rule fires for representative examples before enabling them for production.

</Step>

<Step title="Configure handoff settings and guardrails" icon="shield">

Switch to the **Handoff Settings** tab to control how handoffs behave.

- Enable agent handoff and choose a message style that fits your brand.
- Edit the handoff template so it gives humans the right context and sets customer expectations.
- Set guardrails such as maximum handoffs per conversation, cooldowns, and loop prevention.

Start with conservative limits (for example, 3 handoffs per conversation) and only relax them once you see stable behavior.

</Step>

<Step title="Test flows and roll out gradually" icon="play-circle">

Validate routing and handoffs before sending all traffic through them.

- Use test numbers, email aliases, or staging channels to exercise each rule and guardrail.
- Confirm that the correct rule matches, guardrails behave as expected, and the right assistant or human receives the conversation.
- Roll out in stages (by channel, segment, or percentage) and monitor escalation rates and operator feedback.

Adjust rules, templates, and guardrails as needed, then expand coverage once results are predictable.

</Step>

</Steps>

## Routing rules vs handoff settings

Use the tabs below to mirror how the Routing & Handoffs UI is organized.

<Tabs>

<Tab title="Routing Rules" icon="code">

Routing Rules define when the system should automatically transfer a conversation to a different assistant based on message content, intent, and channel.

### Routing rule fields

Use these fields when creating or editing a routing rule.

<ParamField body="ruleName" param-type="string" required="true" show-location="false">

**Rule Name** shown in the Routing Rules list. Choose a descriptive name like `Billing chat handoffs` or `Technical email escalations` so operators can quickly understand what the rule does.

</ParamField>

<ParamField body="triggers.keywords" param-type="string (comma-separated)" required="false" show-location="false">

**Triggers · Keywords** that appear in the conversation content and should activate the rule. Enter a comma‑separated list such as `invoice, refund, overcharged`. Matching is typically case‑insensitive; use specific phrases to avoid accidental matches.

</ParamField>

<ParamField body="triggers.intent" param-type="enum" required="false" show-location="false">

**Triggers · Intent** selected from a dropdown of high‑level intents. Options include `None`, `Sales`, `Support`, `Billing`, `Technical`, and `General`. Choose an intent when you want the rule to activate based on classified intent instead of or in addition to raw keywords.

</ParamField>

<ParamField body="targetAssistant" param-type="assistant reference" required="true" show-location="false">

**Target Assistant** that should receive the conversation when the rule fires. Select from your configured assistants (for example, Billing assistant, Technical assistant, Sales assistant). The transfer will route the conversation to this assistant if guardrails allow it.

</ParamField>

<ParamField body="channels" param-type="multi-select enum" required="true" show-location="false">

**Channels** where this rule applies. Choose one or more of `SMS`, `Chat`, `Voice`, and `Email`. The rule only evaluates traffic from the selected channels, so you can use separate rules per channel when needed.

</ParamField>

<ParamField body="priority" param-type="integer" required="true" show-location="false">

**Priority** that controls which rule runs first when multiple rules match. Higher‑priority rules are evaluated before lower‑priority ones. Use this to ensure specific rules (for example, Enterprise billing) override broader ones (for example, General billing).

</ParamField>

<ParamField body="status" param-type="enum" required="true" show-location="false">

**Status** of the rule: `Active` or `Inactive`. Active rules participate in routing. Inactive rules are saved for later and do not affect traffic, which is useful for staging or temporarily disabling a path without deleting it.

</ParamField>

### Common routing rule patterns

- Route **Billing** intent on Chat and Email to a dedicated billing assistant with tighter guardrails.
- Route **Sales** intent on Voice to a sales assistant that focuses on call handling.
- Route **Technical** intent on SMS to a concise troubleshooting assistant optimized for short messages.
- Use different priorities to ensure high‑value customer segments are routed before general traffic rules.

</Tab>

<Tab title="Handoff Settings" icon="users">

Handoff Settings control whether agent handoffs are enabled, how the system announces the transfer, and which guardrails apply before any conversation is moved.

### Core handoff settings

These settings apply to all routing‑driven transfers.

<ParamField body="enableAgentHandoff" param-type="boolean" required="true" show-location="false">

**Enable Agent Handoff** toggle that turns automatic handoffs on or off. When disabled, routing rules can still match but the system will not transfer conversations to human agents or other assistants through the handoff engine.

</ParamField>

<ParamField body="messageStylePreset" param-type="enum" required="true" show-location="false">

**Message Style preset** that controls tone and length of the handoff message. Available presets include `Default`, `Professional`, `Casual`, `Brief`, and `Detailed`. Use `Professional` for support queues, `Casual` for community or low‑stakes channels, and `Brief` for SMS or tight UIs.

</ParamField>

<ParamField body="handoffMessageTemplate" param-type="string (template)" required="true" show-location="false">

**Handoff Message Template** shown to the customer when a handoff occurs. The template supports placeholders in curly braces:

- `agent_name` – the human or assistant taking over.
- `task_description` – a short summary of what needs to be done.
- `customer_intent` – the detected intent such as Billing or Support.

For example: `I am connecting you with {agent_name} from our {customer_intent} team to help with {task_description}.`

</ParamField>

<ParamField body="humanResponseTime" param-type="string or numeric estimate" required="false" show-location="false">

**Human Response Time** expectation shown to the customer, such as `a few minutes` or `within 15 minutes`. Set this to match your actual SLAs so customers know how long to wait after a handoff.

</ParamField>

### Handoff guardrails

Guardrails define the safety limits that every potential handoff must pass before the system executes it.

<ParamField body="maxHandoffsPerConversation" param-type="integer (1–10)" required="true" show-location="false">

**Maximum Handoffs Per Conversation** sets a hard cap on how many times a single conversation can be transferred. Valid values range from `1` to `10`; a practical starting point is **3–5**. After the limit is reached, additional handoff attempts are blocked.

</ParamField>

<ParamField body="cooldownMinimumMessages" param-type="integer" required="false" show-location="false">

**Cooldown Minimum Messages** defines how many new messages must occur between handoffs in the same conversation. Use this to ensure the assistant or agent has a chance to make progress before another transfer is allowed.

</ParamField>

<ParamField body="cooldownMinimumMinutes" param-type="integer (minutes)" required="false" show-location="false">

**Cooldown Minimum Minutes** defines the minimum time that must pass between handoffs. Combine this with message‑based cooldowns when you want to limit both rapid back‑and‑forth and long‑running ping‑pong behavior.

</ParamField>

<ParamField body="preventHandoffLoops" param-type="boolean" required="true" show-location="false">

**Prevent Handoff Loops** option that blocks repetitive cycles such as `A → B → A`. When enabled, the system tracks recent transfers in a conversation and refuses to execute a handoff that would send ownership back to a recently involved assistant in a loop.

</ParamField>

<ParamField body="maxHandoffsPerHour" param-type="integer" required="false" show-location="false">

**Maximum Handoffs Per Hour** sets an optional rate limit on handoffs across your tenant or per conversation, depending on implementation. Use this to protect human queues from sudden spikes in automated escalation.

</ParamField>

<Callout kind="info">

Start with stricter guardrails than you think you need, monitor behavior for a few days, then relax limits if you see stable, high‑quality handoffs and low operator frustration.

</Callout>

</Tab>

</Tabs>

## Flow: from rule match to transfer

Every potential handoff follows the same high‑level flow:

1. **Routing rule matches**  
   The system evaluates active Routing Rules for the current channel and conversation. When a rule matches its triggers (keywords and/or intent), it proposes a transfer to the rule's target assistant.

2. **Guardrails check in Handoff Settings**  
   Before moving the conversation, the handoff engine checks all configured guardrails: maximum handoffs per conversation, cooldown messages and minutes, loop prevention, and hourly limits. If any guardrail fails, the handoff is blocked.

3. **Message and transfer**  
   If all guardrails pass and agent handoff is enabled, the system renders the handoff message template using placeholders like `agent_name`, `task_description`, and `customer_intent`, sends it to the customer, and transfers ownership to the target assistant or human queue.

A well‑tuned configuration keeps most conversations on the best‑suited assistant while ensuring humans see only the escalations that matter and never receive repeated or looping transfers.

## Operate and iterate safely

Once routing rules and handoff settings are live, treat them as an operational control surface, not a one‑time setup.

- **Review metrics by rule and intent**: Track which rules fire most often, how many handoffs guardrails block, and how many escalations resolve successfully.
- **Adjust priorities and triggers**: Tighten or loosen keyword lists and intents when rules match too broadly or too rarely.
- **Review guardrail impacts**: If agents see too many low‑value escalations, make guardrails stricter; if legitimate escalations are blocked, relax limits slightly.
- **Coordinate with training and playbooks**: Update assistants, playbooks, and training context when you introduce new routing paths or escalation policies.

<Callout kind="info">

When you change routing or handoff behavior, update internal runbooks so support and operations teams understand when handoffs will occur, how to override AI if needed, and where to report misrouted conversations.

</Callout>
